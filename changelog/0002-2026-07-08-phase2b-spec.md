## Context
Phase 2a (My Cluster read view) is complete. Each store card on the AM's My
Cluster screen has two title-bar buttons currently stubbed with a "coming next"
snackbar:
- Move staff between stores (swap icon)
- Request staff for this store (person-add icon)

Phase 2b wires these two flows.

## Decisions locked for this build
- **Move destination scope:** an Area Manager may only move a staff member to
  another store that is WITHIN the AM's own assigned brands (their cluster). They
  cannot move staff to stores outside their purview.
- **Move semantics:** "Move" changes the employee's **primary store**
  (`employees.primary_store_id`). It does NOT touch additional-store assignments.
  (Additional-store add/remove is a separate concern, handled by the existing
  employee edit flow / a later phase.)
- **Oversight:** any AM-initiated move creates a **notification to admins**
  (recipient_role = 'Admin') recording who moved whom, from which store to which,
  and when. The move itself is applied immediately (not queued) — the notification
  is for visibility, not approval.
- **Request staff is a QUEUED request, not an immediate assignment.** The AM
  searches by name; on a match they submit a request; this creates a notification
  to admins/IT. An admin performs the actual assignment later (Phase 3 inbox).

## Flow 1 — Move staff between stores
UI (from a store card): tapping Move opens a dialog listing the staff currently
in that store; pick one, then pick a destination store (dropdown limited to the
AM's cluster stores, excluding the current one); confirm.

Backend: `POST /api/v1/employees/{employee_id}/move`
- Body: `{ "to_store_id": int }`
- Auth: Area Manager only (Admin/HR use normal edit).
- Validate: caller is an AM; employee is currently in the AM's cluster (primary
  or additional store within AM brands); `to_store_id` is within the AM's cluster.
- Effect: set `employees.primary_store_id = to_store_id`.
- Write `staff_status_log` row: action_type e.g. "TRANSFER", details
  `{from_store_id, to_store_id, by_user_id}`. (Note: `ALLOWED_STATUS_ACTIONS`
  in models currently is PROMOTION/DEMOTION/TERMINATION — add TRANSFER, or use a
  free-text action; confirm during build.)
- Create `notifications` row: recipient_role='Admin', type='STAFF_MOVED',
  payload `{employee_id, employee_name, from_store, to_store, by_user}`.
- Write an `audit_logs` UPDATE row (consistent with other mutations).
- Return the updated employee (or 204 + client refetch).

## Flow 2 — Request staff for a store
UI (from a store card): tapping Request opens a form with a single "Full name"
field. On submit, query all staff by name (case-insensitive contains). Show
matches, each displaying: name, associated brand(s), and the store(s) they're in.
If the right person is shown, an "Request" button submits the request for THIS
store.

Backend:
- Search: `GET /api/v1/employees/search?name=<q>`
  - Auth: Area Manager (and Admin/HR).
  - Returns minimal cards: `{employee_id, employee_name, brand_names[],
    store_names[]}` — brand derived from primary store's brand + additional
    stores' brands; stores = primary + additional. Cap results (e.g. 25).
  - NOTE: this search is intentionally across ALL staff (not just the AM's
    cluster) — the point is to find someone elsewhere and pull them in.
- Request: `POST /api/v1/employees/{employee_id}/request-assignment`
  - Body: `{ "store_id": int }` (the requesting store, must be in AM's cluster).
  - Auth: Area Manager.
  - Effect: NO change to employee. Create `notifications` row:
    recipient_role='Admin', type='STAFF_REQUESTED',
    payload `{employee_id, employee_name, requested_store_id, requested_store_name,
    by_user_id, by_user_name}`.
  - Write `audit_logs` row (action e.g. INSERT on a logical "requests" concept, or
    a dedicated note) — confirm approach during build.
  - Return 201/204.
